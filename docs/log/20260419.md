## 학습

- **[kube-proxy](../kubernetes/kube-proxy.md)** — Service·Endpoints Watch → iptables 업데이트. 로드밸런싱(확률 가중치), NodePort 처리. ClusterIP/NodePort 규칙이 같은 iptables 체인 공유.
- **[kubelet](../kubernetes/kubelet.md)** — 각 노드의 에이전트. API Server Watch → 자기 노드에 배정된 Pod 감지 → CRI 통해 containerd에 실행 요청. 이름은 kube + -let(작은 ~).
- **[Controller](../kubernetes/controller.md)** — 선언된 상태와 실제 상태를 맞추는 Reconcile Loop. 폴링이 아닌 Watch(이벤트) 방식. Controller는 Pod 오브젝트 생성까지, 실제 실행은 kubelet 담당.
- **[etcd](../kubernetes/etcd.md)** — 쿠버네티스의 데이터베이스. 모든 오브젝트의 선언된 상태 저장. 컴포넌트는 API Server를 통해서만 접근.
- **[API 그룹](../kubernetes/api-group.md)** — api-version.md에서 분리. 코어 리소스에 그룹이 없는 이유(역사적 이유, 하위 호환성).

## 퀴즈

**Q1. Service가 없으면 어떤 문제가 생기고, Service는 어떻게 해결하는가?** — 4/5
> 고정 IP 없이 파드 IP로 직접 통신하면 재시작 시마다 IP가 바뀌어 접근 불가. Service가 ClusterIP/NodePort 같은 고정 IP와 파드 IP를 매핑해 요청을 전달.

부족한 부분: 고정 진입점은 IP뿐 아니라 DNS 이름도 포함됨. Service 생성 시 CoreDNS에도 등록되어 `kafka:9092`처럼 이름으로 접근 가능.

**Q2. `kafka:9092` 요청이 파드 IP에 도달하기까지의 흐름 (DNS → iptables)** — 3/5
> DNS로 IP를 알아낸 뒤 Service에서 실제 파드 IP를 추출한다.

부족한 부분: DNS가 반환하는 건 ClusterIP이지 파드 IP가 아님. ClusterIP → 파드 IP 변환은 Service 오브젝트가 하는 게 아니라 kube-proxy가 등록한 iptables 규칙을 커널이 패킷 레벨에서 NAT 변환하는 것.

**Q3. 같은 `kind: Kafka`를 쓰는 두 오퍼레이터가 공존할 수 있는가? apiVersion의 역할은?** — 3/5
> 오퍼레이터는 apiVersion을 보고 자신이 관리할 수 있는 리소스인지 구분할 수 있다.

부족한 부분: 공존 가능 여부 언급 없음(가능하다). 구분 주체는 오퍼레이터가 아니라 API Server — `apiVersion + kind` 조합으로 어떤 CRD인지 판단하고, 오퍼레이터는 자신이 등록한 CRD를 Watch할 뿐.

**Q3. (재답변) 같은 `kind: Kafka`를 쓰는 두 오퍼레이터가 공존할 수 있는가? apiVersion의 역할은?** — 5/5
> 공존 가능하다. apiVersion은 API Server가 등록된 CRD가 어떤 오퍼레이터가 등록한 것인지 구분할 수 있게 해준다.

**Q4. `apiVersion: apps/v1`과 `apiVersion: v1`의 요청 경로가 다른 이유는?** — 2/5
> 기본 리소스는 v1, CRD는 apps/v1로 표시한다.

부족한 부분: apps/v1은 CRD가 아니라 쿠버네티스 빌트인 API 그룹. 경로가 다른 이유는 코어 리소스(v1)는 그룹이 없어 `/api/v1/...`, 나머지는 그룹이 있어 `/apis/{그룹}/{버전}/...` 경로를 사용하기 때문.

**Q5. 파드가 죽고 새로 떴을 때 클라이언트가 주소를 바꾸지 않아도 되는 이유와 관련 컴포넌트** — 4/5
> 클라이언트는 DNS만 기억하면 됨. DNS에 매핑된 ClusterIP는 iptables에 실제 파드 IP와 매핑되어 동작. 파드가 새로 뜨면 iptables가 새 파드 IP로 갱신됨.

부족한 부분: iptables 갱신 주체 누락. Service가 직접 갱신하는 게 아니라 kube-proxy가 파드 변화를 감지해서 iptables 규칙을 업데이트함.

**Q6. Strimzi 오퍼레이터 설치 시 일어나는 일 두 가지와 kubectl apply 이후 Kafka 프로세스까지의 흐름** — 2/5
> CRD를 API Server에 등록. API Server에 등록된 CRD가 있는지 확인하고 있다면 Controller가 생성함.

부족한 부분: 두 번째 일 누락 — CRD 등록 외에 Controller도 함께 배포됨. kubectl apply 이후 흐름 없음 — API Server가 etcd에 저장 → Controller 감지 → StatefulSet/Service 등 오브젝트 생성 → kubelet 감지 → containerd → Kafka 프로세스 기동.

**Q7. Ingress와 Service의 차이는? Ingress가 없으면 어떤 문제가 생기는가?** — 3/5
> Service는 내부 통신용, Ingress는 외부 통신용. Ingress가 없으면 외부와 통신 불가.

부족한 부분: Service도 NodePort/LoadBalancer 타입으로 외부 접근 가능. Ingress의 핵심은 L7(HTTP) 레이어에서 호스트/경로 기반 라우팅 — 서비스마다 별도 포트 없이 api.example.com, app.example.com처럼 하나의 진입점에서 여러 서비스로 라우팅.

**Q8. Ingress Controller와 Controller의 차이는? Ingress Controller가 하는 일은?** — 3/5
> Ingress Controller는 실제 트래픽을 분산/라우팅. Controller는 API에 따라 상태를 유지.

부족한 부분: Ingress Controller는 Controller 역할(Ingress 리소스 Watch → nginx 설정 갱신)도 겸함. 일반 Controller와 달리 nginx 같은 리버스 프록시를 내장해서 실제 트래픽 경로 위에도 있다는 게 핵심 차이.

**Q9. Container Runtime이란 무엇이고, kubelet과 어떤 관계인가?** — 4/5
> 실제 컨테이너가 동작하는 곳으로 containerd, docker가 있다. kubelet은 컨테이너 런타임을 이용해서 컨테이너를 동작시킨다.

부족한 부분: kubelet과 런타임 사이 CRI(Container Runtime Interface) 누락. kubelet은 CRI 표준을 통해 통신하므로 런타임 교체 가능. Docker는 CRI 미구현으로 dockershim 필요했고 k8s 1.24에서 제거됨.

**Q10. Node란 무엇이고, 컨트롤 플레인 노드와 워커 노드의 차이는?** — 5/5
> 노드는 실제 컴퓨팅 자원. 컨트롤 플레인 노드는 워커 노드의 동작을 관리, 워커 노드는 실제 애플리케이션이 실행되는 곳.
