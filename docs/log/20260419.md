## Kubernetes

**[Kubernetes 핵심 개념](../kubernetes/core.md)**
쿠버네티스의 두 축. API는 원하는 상태를 선언하는 수단(YAML 매니페스트), 클러스터는 그 선언이 실현되는 실행 환경. 개발자는 API로 의도만 표현하고 어느 노드에서 실행할지는 쿠버네티스가 결정.

**[API Server 통신](../kubernetes/api-server.md)**
kubectl → API Server는 HTTPS REST. API Server → etcd는 gRPC. kubelet/kube-proxy/CoreDNS는 API Server를 Watch(롱폴링)하다가 변경이 생기면 반응. API Server가 능동적으로 호출하는 건 etcd와 kubelet뿐.

**[apiVersion URL 매핑](../kubernetes/api-version.md)**
`apiVersion`은 API Server 엔드포인트 경로와 1:1 매핑. 그룹 있으면 `/apis/그룹/버전/...`, 코어 리소스는 `/api/버전/...`. `kubectl apply -f`가 해당 경로로 HTTPS POST 요청을 보내는 것.

**[Ingress Controller vs Controller](../kubernetes/ingress.md)**
일반 Controller(Deployment 등)는 쿠버네티스 오브젝트만 관리. Ingress Controller는 reconciliation loop에 더해 실제 네트워크 트래픽까지 직접 처리하는 데이터 플레인 역할도 겸함.

**[Container Runtime](../kubernetes/container-runtime.md)**
컨테이너 이미지를 읽어 실제 프로세스로 실행하는 프로그램. Docker도 내부적으로 containerd 사용. 쿠버네티스 1.24부터 dockershim 제거하고 containerd 직접 사용. 이미지 포맷이 OCI 표준이라 Docker로 빌드한 이미지를 containerd에서 그대로 실행 가능.

**[Node](../kubernetes/node.md)**
클러스터를 구성하는 서버 하나하나. 컨트롤 플레인 노드(API Server, etcd, Scheduler, Controller Manager)와 워커 노드(kubelet, kube-proxy, containerd, 파드)로 구분. kubelet이 API Server를 Watch하다가 파드 실행 명령을 받으면 containerd에 요청.
